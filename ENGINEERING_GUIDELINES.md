# 🏗 Інженерні стандарти та правила розробки
## PlayPhoto HR Bot (Node.js + TypeScript)

Цей документ визначає суворі технічні стандарти та архітектурні принципи проекту. Ці правила є обов'язковими для всієї генерації коду та ручної розробки.

---

### 1. Архітектура та структура модулів (Запобігання регресіям)

*   **Принцип єдиної відповідальності (SRP):**
    *   `main.ts`: Тільки ініціалізація бота, підключення middleware та глобальних хендлерів. Бізнес-логіка в `main.ts` **заборонена**.
    *   **Правило ізоляції:** Всі нові функції повинні створюватися в окремих файлах у `handlers/` або `services/`. Редагування `main.ts` допустиме тільки для реєстрації нового модуля.
    *   `services/`: Ізольована бізнес-логіка (Google Calendar, обробка зображень).
        *   **Ідемпотентність:** Сервіси повинні бути атомарними. Наприклад, `bookingService.bookSlot` повинен або завершитися успішно (БД + Календар), або відкотити все. Жодних побічних ефектів при помилках.
        *   **Ізоляція контексту:** Сервіси не повинні приймати `ctx`. Тільки необхідні дані (userId, text тощо).
    *   **Правило стеку Middleware:** Порядок реєстрації в `main.ts` є критичним. Завжди:
        1. Глобальні логери/обробники помилок.
        2. Безпека/Захист від флуду.
        3. Сесії та діалоги (Діалоги ПОВИННІ бути вище за `bot.on("message")`).
        4. Меню та плагіни.
        5. Специфічні команди (`/start`, `/admin`).
        6. Глобальні обробники "catch-all" (в останню чергу).

---

*   **Зв'язність (Coupling):**
    *   **Заборонено** циклічні залежності (Circular Dependencies).
    *   Всі модулі повинні використовувати аліаси або відносні шляхи з розширенням `.js` (згідно з `type: module` у `package.json`).

### 2. TypeScript та типізація

*   **Strict Mode:** Використання `any` суворо **заборонено**. Якщо тип невідомий, використовувати `unknown` з подальшим перевіркою типу (type guarding).
*   **Zod Validation:** Всі дані з зовнішніх джерел (env, API, введення від користувача) повинні валідуватися через `zod`.
*   **Типізація контексту:** Завжди використовувати `MyContext` (розширений тип grammY).
*   **Інтерфейси замість типів:** Для опису об'єктів та моделей даних використовувати `interface`.

### 3. Робота з Telegram API (grammY)

*   **Conversations (Stateless):** Використання плагіну `@grammyjs/conversations` **суворо заборонено**, оскільки він споживає багато пам'яті та може створювати проблеми з паралельною обробкою. Для багатокрокових діалогів потрібно використовувати Stateless підхід на базі кінцевих автоматів (State Machine) через `ctx.session.step`.
*   **Меню (grammY/menu):**
    *   **CRITICAL:** Кожне нове меню повинно бути зареєстроване в батьківському меню за допомогою `.register(subMenu)`. Навігація через `ctx.menu.nav("menu-id")` працює лише для зареєстрованих підменю.
*   **Керування сесіями (Lazy Sessions):**
    *   Всі сесії працюють через кастомний `lazySession` middleware, який кешує дані в RAM на короткий час і асинхронно синхронізує їх з Redis (Debouncing).
    *   Сесії використовують `JSON.stringify` з кастомним `bigIntReplacer`, тому збереження `BigInt` (наприклад, Telegram ID) є безпечним.
    *   Дані сесії повинні бути мінімальними. Не зберігати великі об'єкти з БД в сесії, зберігати тільки їх `ID`.
*   **Rate Limiting:** Обов'язкове використання middleware для обмеження частоти запитів (Throttler/Flood-protection).
*   **Callbacks:** Кожен `callback_query` повинен завершуватися викликом `ctx.answerCallbackQuery()`.
*   **Мова та UI (Language & consistency):**
    *   **Адмін-панель (Admin UI):** Всі кнопки, статуси, логи та системні повідомлення для адміністраторів повинні бути **англійською мовою**. Це забезпечує професійний вигляд та технічну чіткість.
    *   **Інтерфейс користувача (Staff/Candidate UI):** Весь контент, орієнтований на фотографів та кандидатів, залишається **українською мовою**.
    *   **Централізація текстів:** Всі текстові константи для адмінки повинні зберігатися в `src/constants/admin-texts.ts`.

### 4. Робота з базою даних (Prisma)

*   **Транзакції:** Всі операції, що включають зміну кількох пов'язаних сутностей (наприклад, бронювання слота та зміна статусу кандидата), **повинні** виконуватися через `$transaction`.
*   **Конкурентність та блокування:** Для операцій бронювання використовувати "атомарні оновлення": `updateMany({ where: { id, isBooked: false }, data: { isBooked: true } })`. Якщо результат 0 — слот вже зайнятий (запобігання стану гонитви).
*   **Міграції:** Будь-які зміни схеми БД виконуються лише через `prisma migrate dev`. Пряме редагування БД заборонено.

### 5. Фонова автоматизація (Workers)

*   **Захист процесів:** Вокери повинні працювати в незалежних циклах. Жодна помилка у вокері не повинна зупиняти основний процес бота.
*   **Ідемпотентні сповіщення:** Кожне нагадування повинно мати прапорець у БД (наприклад, `reminded6h`). Маркувати як "надіслано" ТІЛЬКИ ПІСЛЯ успішного виклику Telegram API.
*   **Стійкість API:** У циклах вокера обов'язковий `try-catch` всередині циклу `for`, щоб одна помилка (наприклад, заблокований бот) не зупинила розсилку іншим.

### 6. Цілісність даних та схема

*   **Enum First:** Всі статуси (`status`), етапи (`step`) та типи даних (наприклад, `city`) повинні бути описані як `enum` у Prisma. Уникати "магічних рядків".
*   **Централізація ролей:** Єдиним джерелом правди про права доступу є `User.role`. Перевірка в хендлерах повинна базуватися на цьому полі.

*   **Global Catch:** `bot.catch` повинен бути налаштований на логування помилок у зовнішню систему (Sentry/Logtail) та повідомлення адмінів про критичні збої.
*   **Поступова деградація (Graceful Degradation):** Помилка в сторонньому сервісі (наприклад, Google Calendar) не повинна призводити до падіння всього бота або блокування діалогу користувача. Використовувати `try-catch` блоки з інформативними повідомленнями для користувача.
*   **Логування:**
    *   `console.log` дозволений тільки для налагодження.
    *   Для продакшену використовувати структуроване логування (рівні: `info`, `warn`, `error`) через `src/core/logger.ts`.
    *   **СУВОРО ЗАБОРОНЕНО** створювати власні лог-файли (наприклад, `debug.log`, `bot.log`) у корені проекту. Всі логи повинні йти в `stdout`, щоб ними керував Docker.
    *   Якщо потрібно логування у файл для специфічних потреб, використовувати папку `/logs`, яка має бути в `.gitignore` та монтуватися як volume.

### 7. Безпека та масштабованість

*   **Secrets:** Жодних жорстко закодованих токенів, ID адмінів чи паролів. Все через `.env`.
*   **Очищення введення (Input Sanitization):** Всі текстові введення користувачів повинні очищуватися перед використанням у SQL (Prisma це робить автоматично) або HTML-повідомленнях (`parse_mode: "HTML"`).
*   **Stateless Services:** Сервіси повинні бути спроектовані так, щоб бот міг працювати у кількох інстансах (горизонтальне масштабування). Стан зберігається тільки в БД/Redis, а не в пам'яті процесу.

### 8. Приклад правильної структури обробника (Стандартний патерн)

```typescript
// src/handlers/example.ts
import { MyContext } from "../types/context.js";
import { bookingService } from "../services/booking.js";

export async function handleBooking(ctx: MyContext) {
    try {
        const slotId = ctx.match?.[1];
        if (!slotId) throw new Error("Missing slotId");

        await ctx.answerCallbackQuery("Бронюємо...");
        
        // Виклик сервісу, а не пряма робота з prisma
        const result = await bookingService.bookSlot(slotId, ctx.from!.id);
        
        await ctx.reply(`✅ Успішно! Ваш час: ${result.time}`);
    } catch (error) {
        console.error("Booking error:", error);
        await ctx.reply("❌ Не вдалося забронювати. Спробуйте інший слот.");
    }
}
```

### 10. Правила запобігання регресіям (Чек-лист)

Перед кожною зміною чи комітом слід перевірити:
1.  **Shadowing:** Чи не перехоплює мій новий обробник повідомлення, які призначені для вже існуючих діалогів (Conversation) або меню? Уникати загальних `bot.on("message")` без фільтрації за станом сесії.
2.  **Забруднення стану (State Pollution):** Чи не змінюю я `ctx.session.step` так, що це може зламати інші сценарії? Уважно контролюйте переключення кінцевого автомата та завжди скидайте стан у `"idle"` (`ctx.session.step = "idle"`) при завершенні або скасуванні кроків.
3.  **Циклічні залежності:** Чи не створив я імпорт, який посилається назад на `main.ts`?
4.  **Приховані припущення:** Чи не покладаюся я на те, що дані вже є в сесії, не перевіряючи їх? Завжди додавати логіку відкату (fallback).
5.  **Побічні ефекти:** Чи не змінює мій код глобальні змінні (як `messageCounts` у `main.ts`) без врахування багатопоточності/масштабування?
6.  **Блокування вводу-виводу (I/O Blocking):** Чи не виконую я довгі мережеві запити (Google Calendar) без попереднього підтвердження користувачу (`ctx.answerCallbackQuery`)?
7.  **Перевірка Enum:** Чи відповідають нові статуси `enum` в `schema.prisma`?

### 11. Дизайн-принципи (надихнуто Apple)

Для забезпечення преміального користувацького досвіду (Premium UX), кожен інтерфейс та меню повинні слідувати принципам Apple Human Interface Guidelines:

*   **Clarity (Чіткість):** Текст та кнопки повинні бути зрозумілими з першого погляду. Використовувати лаконічні підписи (`👥 Team` замість `Список співробітників`).
*   **Hierarchy (Ієрархія):** Найважливіші дії — на першому екрані. Другорядні або специфічні функції — в підменю.

*   **Aesthetic Integrity (Естетична цілісність):** Використовувати єдиний набір емодзі для позначення статусів та категорій. Весь бот повинен відчуватися як єдиний продукт.
*   **Conciseness (Лаконічність):** Головне меню повинно бути максимально компактним. Будь-яка кнопка, якою не користуються кожні 5 хвилин, повинна бути переміщена в підменю "System" або "Ops".

### 13. Інтернаціоналізація (I18n)

Всі користувацькі інтерфейси повинні підтримувати багатомовність через систему **Fluent (.ftl)**.

*   **Заборона Hardcoded Strings:** Будь-який текст, який бачить користувач, повинен бути винесений у файл локалізації `src/locales/uk.ftl`.
*   **Використання `ctx.t()`:** В обробниках та меню використовувати метод `ctx.t("key", { params })` для отримання перекладу.
*   **Типізація FluentVariable:** Fluent приймає лише `string`, `number` або `Date`. Для `bigint` (наприклад, `userId`), `null` або `undefined` обов'язкова конвертація:
    ```typescript
    ctx.t("welcome", { 
        name: user.name || "", // Запобігання null
        id: user.id.toString() // Конвертація BigInt
    });
    ```
*   **Динамічні меню:** У `range.text()` використовувати `ctx.t()` для назв кнопок. Якщо ключ залежить від даних, переконатися, що ключ існує в `.ftl`.
*   **Структура ключів:** Використовувати префікси для групування:
    *   `hr-btn-*` — кнопки HR меню.
    *   `hr-info-*` — інформаційні повідомлення HR.
    *   `screening-ask-*` — питання під час скринінгу.
    *   `support-ans-*` — відповіді в системі підтримки.

### 16. Workflow розробки та деплою (Git Flow)

#### Принцип: `main` = продакшен. Кожен merge в `main` = деплой на сервер.

**Гілки:**
- `main` — тільки стабільний, протестований код. Пуш сюди = автоматичний деплой.
- `dev` — основна гілка розробки. Можна пушити часто — деплою немає, тільки build + тести.
- `feature/назва` (опційно) — окремі великі фічі.

**Робочий процес:**

```
1. Нова задача:
   git checkout dev           # перейти на dev
   # ... вносити зміни ...
   git add src/...            # додавати тільки потрібні файли
   git commit -m "feat: опис"
   git push origin dev        # пуш в dev — НЕ деплоїться

2. Коли готово до релізу:
   git checkout main
   git merge dev
   git push origin main       # ← ось тут деплой відбувається
```

**Що просити у LLM при розробці:**

| Задача | Команда LLM |
|--------|-------------|
| Нова фіча / виправлення | "Зроби зміни, але НЕ пушь" |
| Готово, хочу деплоїти | "Зроби merge dev → main і пуш" |
| Додати поле в БД | "Зміни schema.prisma і створи міграцію (`prisma migrate dev --name назва`), НЕ пушь" |
| Переглянути що змінилось | "Покажи git diff або git log" |

**Правило для LLM:** ніколи не пушити самостійно без явної команди "запуш" або "задеплой". Commit — ок. Push в `dev` — тільки якщо попросили. Push в `main` — тільки якщо явно сказано "деплой в прод".

#### Міграції бази даних

При додаванні полів/таблиць:
1. Змінити `prisma/schema.prisma`
2. Виконати `npx prisma migrate dev --name назва_змін` — створюється файл в `prisma/migrations/`
3. Закомітити разом з кодом
4. При деплої CI/CD автоматично виконає `prisma migrate deploy` на сервері
5. **Контейнер PostgreSQL перезапускати не потрібно** — дані зберігаються у volume

#### Після деплою (рестарт не потрібен якщо):
- Додали/змінили поля в БД — міграція застосовується автоматично
- Змінили логіку бота — новий Docker image запускається автоматично
- **Контейнер `postgres` не перезапускається** при деплоях бота — це окремий сервіс з persistent volume

### 15. Docker та Конфігурація (Золоті правила)

*   **Localhost у Docker:**
    *   **СУВОРО ЗАБОРОНЕНО** використовувати `localhost` або `127.0.0.1` у змінних оточення (`DATABASE_URL`, `REDIS_URL`), які передаються в контейнер.
    *   Для зв'язку між контейнерами **ЗАВЖДИ** використовувати назви сервісів: `postgres`, `redis`.
*   **env_file vs environment:**
    *   Файл `.env` призначений для **локальної розробки** (на хості).
    *   У `docker-compose.yml` змінні підключення до БД/Redis повинні бути явним чином перевизначені в секції `environment`, щоб перекрити значення з `.env`.
    *   Приклад:
        ```yaml
        environment:
          - DATABASE_URL=postgresql://user:pass@postgres:5432/db # Правильно (ім'я сервісу)
        env_file:
          - .env # Може містити DATABASE_URL=...localhost... (для локального запуску)
        ```

### 17. 🏗️ Архітектура 10/10 (SMI + Smart Navigation)

Для забезпечення преміальної якості (Apple Style) та технічної стабільності, ми використовуємо наступні стандарти:

1.  **Принцип SMI (Single Message Interface):** Бот взаємодіє з користувачем через ОДНЕ повідомлення. Всі переходи — це `editMessageText`.
2.  **ScreenManager:** Заборонено використовувати `ctx.editMessageText` або `ctx.reply` напряму в меню. Тільки `ScreenManager.renderScreen(ctx, text, menuObject, { pushToStack: true })`.
3.  **Smart Back Button:** Для повернення назад використовувати `ScreenManager.goBack(ctx, fallbackText, fallbackMenu)`. Це автоматично працює через `navStack` у сесії.
4.  **Zero Circular Dependencies:** Всі меню одного модуля (наприклад, HR) повинні знаходитись в ОДНОМУ файлі, щоб уникнути циклічних імпортів.
5.  **Safe Edit:** ScreenManager автоматично ігнорує помилку "message is not modified", якщо контент не змінився. Розробнику не потрібно про це турбуватися.
6.  **Чистота чату (Conversations/Input):** Якщо користувач вводить текст, хендлер зобов'язаний видалити вхідне повідомлення користувача (`ctx.deleteMessage()`) відразу після зчитування даних, щоб зберегти SMI.

### 18. 🛡️ Безпека розсилок та автоматичних сповіщень (Broadcast Safety)

Щоб уникнути випадкових масових розсилок (особливо діючому персоналу) та забезпечити надійність системи, впроваджено наступні правила:

1.  **Сувора фільтрація за статусом (Explicit Status Filtering):**
    Будь-яка функція, що надсилає автоматичні вітання (Welcome) або сповіщення про готовність графіка, **ПОВИННА** містити явний фільтр за статусом кандидата.
    *   **Приклад:** При надсиланні вітання новому фотографу обов'язково перевіряти `candidate.status === 'AWAITING_FIRST_SHIFT'`.
    *   **Заборонено** покладатися лише на прапорці типу `isWelcomeSent: false`, оскільки для існуючих співробітників вони можуть бути не проставлені.

2.  **Простежуваність сповіщень (Traceability & Rollback):**
    При відправці системних сповіщень через `api.sendMessage`, результат (включаючи `message_id`) повинен бути зафіксований (наприклад, у `ChatLog`). Це дозволить у разі помилки створити скрипт для масового видалення помилкових повідомлень через `deleteMessage`.

3.  **Подвійна перевірка (Double-check) у циклах:**
    У циклах розсилки (наприклад, при синхронізації `Full Sync`) перед кожним викликом `api.sendMessage` слід ще раз перевірити статус об'єкта користувача, щоб уникнути відправки повідомлення "застарілому" списку.

### 19. 🗄 Database Schema Change Policy (Запобігання Schema Drift)

Цей розділ визначає суворий порядок дій при зміні структури бази даних. Будь-яке порушення призводить до **Prisma SCHEMA DRIFT**, зупинки CI та блокування деплою.

#### 1. Золоте правило атомарності
**Заборонено змінювати `schema.prisma` без одночасного створення міграції.**
Будь-який комміт, що містить зміни у файлі `prisma/schema.prisma`, ПОВИНЕН також містити:
*   Нову папку у `prisma/migrations/YYYYMMDDHHMMSS_name/` з файлом `migration.sql`.
*   Оновлений файл `prisma/migration_lock.toml`.

#### 2. Процедура внесення змін (Step-by-step)
1.  **Редагування схеми:** Внесіть зміни в `prisma/schema.prisma`.
2.  **Генерація міграції:** Запустіть CLI Prisma. Для цього ОБОВ'ЯЗКОВО мати підключення до робочої (або Shadow) бази даних:
    ```bash
    npx prisma migrate dev --name <опис_зміни>
    ```
3.  **Валідація артефактів:** Перевірте, що Prisma створила SQL-файл. Переконайтеся, що SQL відповідає вашим очікуванням.
4.  **Атомарний комміт:** Додайте всі файли (`schema.prisma`, `migrations/`, `migration_lock.toml`) в один комміт.

#### 3. Суворі заборони
*   🚫 **Заборонено ручне створення міграцій:** Ніколи не створюйте папки migrations або SQL-файли вручну через `mkdir` або `cat`. Тільки через CLI Prisma.
*   🚫 **Заборонено прямі зміни через SQL-клієнти:** Жодних `ALTER TABLE` через DBeaver/DataGrip на Dev/Staging/Prod. Всі зміни — тільки через код.
*   🚫 **Заборонено `prisma db push` для продакшну:** Ця команда ігнорує історію міграцій і гарантовано ламає CI.

#### 4. Правила для AI-агентів (Gemini/Copilot)
Якщо ви дієте як AI-агент:
*   Ви **не маєте права** вносити зміни в `schema.prisma`, якщо у вас немає доступу до працюючої бази даних для виконання команди `migrate dev`.
*   Якщо БД недоступна, ви **зобов'язані** попередити користувача:
    > "⚠️ Я можу змінити файл схеми, але не можу згенерувати валідну міграцію (БД недоступна). Будь ласка, запустіть `npx prisma migrate dev` самостійно після моїх змін, щоб уникнути SCHEMA DRIFT."
*   Ніколи не намагайтеся "підробити" SQL-файли міграцій.

#### 5. CI Перевірка на Drift
У нашому CI (GitHub Actions) працює автоматична перевірка:
```bash
npx prisma migrate diff --from-schema-datasource prisma/schema.prisma --to-schema-datamodel prisma/schema.prisma --exit-code
```
Якщо ця команда повертає помилку — комміт буде відхилено. Це означає, що стан схеми не відповідає історії міграцій у репозиторії.
